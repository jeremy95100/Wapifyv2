### 3. Corrections automatiques du code généré

#### a) Apostrophes typographiques → ASCII
**Fonction :** `fixTypographicApostrophes()` ([lib/react-generator.ts:1693-1732](lib/react-generator.ts#L1693-L1732))

Remplace automatiquement :
- `'` (U+2019) → `'` (U+0027)
- `'` (U+2018) → `'` (U+0027)

**Résultat :** 899 apostrophes corrigées automatiquement lors d'une génération test.

#### b) Props `asChild` invalides
**Fonction :** `removeAsChildProp()` ([lib/react-generator.ts:1756-1797](lib/react-generator.ts#L1756-L1797))

Supprime automatiquement `asChild` prop qui n'existe pas dans notre composant Button.

**Erreur corrigée :**
```
Property 'asChild' does not exist on type ButtonProps
```

#### c) Checkbox `value` → `checked`
**Fonction :** `fixCheckboxValueToChecked()` ([lib/react-generator.ts:1804-1842](lib/react-generator.ts#L1804-L1842))

Corrige automatiquement :
```tsx
// Avant (incorrect)
<input type="checkbox" value={boolean} />

// Après (correct)
<input type="checkbox" checked={boolean} />
```

**Erreur corrigée :**
```
Type 'boolean' is not assignable to type 'string | number | readonly string[]'
```

## 🎯 Sujets en stand-by (À décider)

### 1. Intégration Supabase (Base de données)

**Objectif :** Apps générées avec vraie base de données PostgreSQL

**Workflow proposé :**
```
User s'inscrit → Projet Supabase créé en background
   ↓
User génère app → Claude génère schéma SQL
   ↓
Tables créées automatiquement dans Supabase
   ↓
Code React généré avec @supabase/supabase-js
   ↓
App 100% fonctionnelle avec persistance des données
```

**Architecture retenue :**
- **1 seul projet Supabase** pour tous les utilisateurs (Option B)
- Isolation via Row Level Security (RLS) + `user_id` + `project_id`
- **Coût :** $25/mois maximum (vs $2,500 avec 1 projet/user)
- **Avantages :** Simplicité, scalabilité, rentabilité

**Credentials disponibles :**
- ✅ `NEXT_PUBLIC_SUPABASE_URL` (déjà dans .env.local)
- ✅ `NEXT_PUBLIC_SUPABASE_ANON_KEY` (déjà dans .env.local)
- ✅ `SUPABASE_SERVICE_ROLE_KEY` (déjà dans .env.local)

**Implémentation estimée :** 3-4 heures
- Setup fonctions Supabase
- Génération schéma SQL dans prompt
- Exécution migrations automatiques
- Injection client dans code généré

### 2. Scale (Capacité de génération)

**Situation actuelle :**
- Generate queue : **3 workers** simultanés
- Build queue : **5 workers** simultanés

**Scénarios de charge :**

| Utilisateurs | File d'attente | Temps d'attente max |
|--------------|----------------|---------------------|
| < 10         | 0-3            | 0 min               |
| 10-50        | 3-15           | 3-6 min             |
| 50-100       | 15-30          | 10-20 min           |
| 100+         | 30+            | 2+ heures           |

**Solutions de scale :**

#### Option A : Scale vertical (simple)
- Augmenter `concurrency: 3` → `5-7` workers
- **Coût :** Gratuit (dans limites Railway)
- **Implémentation :** 5 minutes

#### Option B : Scale horizontal (pro)
- 2-3 instances Railway identiques
- Même Redis queue partagée
- **Total :** 9-15 workers simultanés
- **Coût :** $10-15/mois
- **Implémentation :** 30 minutes

#### Recommandation :
**Attendre et monitorer** la charge réelle avant de scaler.
Scale prématuré = gaspillage de ressources.
